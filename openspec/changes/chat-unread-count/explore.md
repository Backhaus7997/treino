## Decision 1 — Where to store read-state

### Option A: `lastRead` map field on the chat doc
Shape: `chats/{chatId}.lastRead: { uid: Timestamp }`. Both members write their own key on read.

- **Pros**: Zero new collections. Client reads state from the same doc already watched by `watchChatsForUser` — no extra Firestore listeners. Simple to implement.
- **Cons**: Security rule challenge — current `allow update` lets a member write any non-`members`/non-`createdAt` field. A user could overwrite `lastRead.otherUid`. Firestore rules cannot inspect map-key granularity with `request.resource.data.lastRead[request.auth.uid]` natively — you'd need a full diff check (`request.resource.data.lastRead.diff(resource.data.lastRead).affectedKeys().hasOnly([request.auth.uid])`). This is possible but non-trivial.
- **Effort**: Low-Medium (rule tightening is the main complexity)

### Option B: Per-user subcollection `chats/{chatId}/reads/{uid}`
Shape: doc per user, `{ lastReadAt: Timestamp }`. Members write only their own doc.

- **Pros**: Security is trivial — `allow write: if request.auth.uid == uid` on the subcollection. Clean isolation. No cross-user tampering possible.
- **Cons**: Requires a separate Firestore read/listener per conversation to get the user's own read-state (can't piggyback `watchChatsForUser`). For a total unread count across N chats: N extra listeners. At small scale (typical 1:1 trainer/athlete) this is acceptable (each trainer has 5–30 athletes), but it's more Firestore listeners.
- **Effort**: Medium (extra listener, extra subcollection)

### Option C: Denormalized per-user unread counter
Shape: `userUnread/{uid}: { totalUnread: int, chatUnread: { chatId: int } }` or per-chat doc. Maintained by Cloud Function on message create; reset by client on read.

- **Pros**: Total badge requires one document read, not N listeners. Scales well.
- **Cons**: Cloud Function complexity. Counter correctness depends on CF not crashing. Race condition if CF fires after client resets counter. Harder to test (CF + Firestore).
- **Effort**: High

### **Recommendation: Option A (map field on chat doc) with rule tightening**

Rationale: the chat doc is already watched by `watchChatsForUser` — adding `lastRead` to it means zero extra Firestore listeners. The `diff().affectedKeys().hasOnly([uid])` rule pattern is supported by Firestore and the project already uses moderately complex rules. For an MVP with 1:1 chats and small user counts (trainers with up to ~30 athletes), this is the right tradeoff.

The rule to add inside `allow update` (replace the current flat check with an augmented one):
```
&& (
  !request.resource.data.keys().hasAny(['lastRead'])
  || request.resource.data.lastRead.diff(resource.data.lastRead).affectedKeys()
       .hasOnly([request.auth.uid])
)
```

## Decision 2 — How to compute unread count

### Option A: Client-side from `lastRead` map vs last message timestamp
Compute: if `chat.lastMessageSenderId != currentUid` AND `chat.lastMessageAt > chat.lastRead[currentUid]` → mark as "has unread". Can show a dot (has/has-not) but not an exact count without querying messages.

- **Pros**: Zero extra queries. Everything is in the chat doc already watched. Works for the "has unread" dot.
- **Cons**: Doesn't give an exact count. For a badge number on the chat icon in the feed header, you'd need to count messages after `lastRead[uid]`, which requires querying messages.

### Option B: Firestore `count()` aggregation query
Per chat: `chats/{chatId}/messages where createdAt > lastRead[uid]`.

- **Pros**: Exact count. No Cloud Function.
- **Cons**: One aggregation query per chat. `fake_cloud_firestore` v2 supports `count()` — testable. Still requires knowing `lastRead[uid]` first. Re-runs on every open. Not real-time (point-in-time).

### Option C: Cloud-Function-maintained counter (`unread.{uid}` field on chat doc)
CF increments `unread.{uid}` for each recipient on message create. Client resets to 0 on open.

- **Pros**: Real-time counter, exact. No client aggregation.
- **Cons**: CF complexity. Counter reset is a separate client write. Race condition: user opens chat while CF is still processing → brief wrong count. More infra.

### **Recommendation: Hybrid — dot + capped count**

For MVP correctness and simplicity:

1. **Per-conversation indicator**: compute client-side from `lastRead` map. If `lastMessageAt > lastRead[currentUid]` AND `lastMessageSenderId != currentUid` → show a dot (or "N" if we want a number). A dot is sufficient for the list row — we don't need an exact count per conversation.
2. **Total feed badge**: count chats where the above is true. This is derived from `List<Chat>` already in `chatsForCurrentUserProvider` — pure Dart computation, no extra Firestore reads.

This means: **no aggregation queries, no CF changes**. Just client-side computation over the already-live stream.

The tradeoff: we show "has unread" (boolean per chat), not "3 unread messages in this chat". This is consistent with WhatsApp's inbox behavior and sufficient for the goal (user sees there's something new and opens the app).

If exact counts per chat are needed in the future, Option B can be layered on top.

## Decision 3 — Mark-as-read trigger

**When**: `ChatScreen.initState` (or first frame post-init) — call `markAsRead(chatId, currentUid)`. Also call it when a new message arrives via `messagesProvider` while the screen is still mounted (use `ref.listen` on `messagesProvider`).

**How**: `ChatRepository.markAsRead(chatId, uid)` — `_chats.doc(chatId).update({'lastRead.$uid': FieldValue.serverTimestamp()})`.

**Edge case — already in chat, new message arrives**: `ref.listen(messagesProvider(chatId), ...)` in `ChatScreen` → on each new snapshot (data), call `markAsRead` again. This keeps the timestamp current without user action.

**Note**: calling `markAsRead` on open is idempotent — if `lastRead[uid]` is already >= `lastMessageAt`, the unread flag is already false. The write still happens but produces no observable change.

## Decision 5 — Security rule (exact shape)

Current `allow update` check (lines 436–439):
```
allow update: if request.auth != null
              && request.auth.uid in resource.data.members
              && request.resource.data.members == resource.data.members
              && request.resource.data.createdAt == resource.data.createdAt;
```

With `lastRead` map field added, tighten to also require that the write only affects the caller's own key:
```
allow update: if request.auth != null
              && request.auth.uid in resource.data.members
              && request.resource.data.members == resource.data.members
              && request.resource.data.createdAt == resource.data.createdAt
              && (
                !request.resource.data.keys().hasAny(['lastRead'])
                || request.resource.data.get('lastRead', {})
                     .diff(resource.data.get('lastRead', {}))
                     .affectedKeys()
                     .hasOnly([request.auth.uid])
              );
```

This allows:
- Sending a message (updates `lastMessageAt`, `lastMessageText`, `lastMessageSenderId`) — `lastRead` not touched, first branch of `||` is true.
- Marking as read (updates `lastRead.{uid}`) — only the caller's own key is in `affectedKeys()`.

A user CANNOT update `lastRead.{otherUid}` because `affectedKeys()` would contain the other user's uid, which is not `hasOnly([request.auth.uid])`.

**Note on `resource.data.get('lastRead', {})`**: `get()` is necessary because `resource.data.lastRead` will throw if the field doesn't exist on legacy chat docs. Using `get('lastRead', {})` is the Firestore rules equivalent of a null-safe access.

## Risks & Unknowns

1. **`fake_cloud_firestore` supports `FieldValue.serverTimestamp()` in updates** — confirmed by existing tests using it in `sendMessage`. The `markAsRead` update will work in tests.
2. **Firestore rules `diff().affectedKeys()`** — this is a production Firestore rules feature; `fake_cloud_firestore` does NOT enforce security rules, so tests won't catch rule bugs. Rules must be tested in the Firebase emulator or via `firestore-rules-unit-testing` (not present in this project). This is a known gap — the rule logic itself must be carefully reviewed.
3. **Race condition on mark-as-read**: if the user opens `ChatScreen`, `markAsRead` fires, but `chatsForCurrentUserProvider` hasn't re-emitted yet, the badge stays for a brief moment. Acceptable for MVP — the stream update is typically sub-second on a real connection.
4. **Legacy chat docs without `lastRead` field**: `chat.lastRead` will be `null` for all existing chats. The unread computation must treat `null` as "never read" — any chat with a `lastMessageSenderId != currentUid` will appear as unread on first app open after deploy. This is correct behavior (user genuinely may not have read it).
5. **watchMessages limit 50**: the unread computation uses `lastMessageAt` vs `lastRead[uid]` (timestamps), NOT message count. The 50-message limit is irrelevant for this approach — we never count individual unread messages client-side.
6. **`Chat` freezed regeneration**: adding `lastRead: Map<String, DateTime>?` requires running `flutter pub run build_runner build` to regenerate `chat.freezed.dart` and `chat.g.dart`. The `TimestampConverter` used for `DateTime` fields needs to handle `Map<String, Timestamp>` deserialization — may need a custom converter. This is a non-trivial technical step.
7. **Timestamp map deserialization**: `Map<String, DateTime>` annotated with a converter on a freezed field is non-standard. The more practical shape may be `Map<String, Timestamp>` stored raw in Firestore and deserialized manually in `_chatFromDoc`, keeping `Chat.lastRead` as `Map<String, DateTime>?` via a dedicated converter. Alternatively, store as `Map<String, int>` (millisecondsSinceEpoch) to avoid converter complexity — but this is less idiomatic.
8. **Notification-side counter**: the existing `notify-chat-message.ts` CF does not need changes — it already sends a push on every message. The badge will appear when the user opens the app and the stream emits.
