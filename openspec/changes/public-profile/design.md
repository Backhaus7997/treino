## 2. Widget API surfaces

All widgets live in the `feed` feature. `PublicProfileScreen` is the only `ConsumerWidget` at the screen level. `PublicProfileFollowButton` is also a `ConsumerWidget` (needs `ref.invalidate` after mutation). The rest are `StatelessWidget` — they receive their data via constructor params (one-way down). This mirrors the `feed-shell-amigos` shape (`FeedScreen` is the orchestrator; cards are dumb).

### 2.1 `PublicProfileScreen`

```dart
class PublicProfileScreen extends ConsumerWidget {
  const PublicProfileScreen({super.key, required this.targetUid});

  final String targetUid;

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

**Rationale**:
- `targetUid` is required and the SOLE constructor param. The route builder passes it from `state.pathParameters['uid']!`.
- `ConsumerWidget` because the screen watches exactly one provider — `publicProfileViewProvider(targetUid)` — and routes `AsyncData/Loading/Error` to the appropriate subtree.
- No `Scaffold` / `AppBackground` / `SafeArea` in this widget — `_ShellScaffold` in `router.dart` already provides them (REQ-PROFILE-SCREEN-001, SCENARIO-207).

### 2.2 `PublicProfileHero`

```dart
class PublicProfileHero extends StatelessWidget {
  const PublicProfileHero({super.key, required this.view});

  final PublicProfileView view;
}
```

**Rationale**:
- `view` is required. The hero needs `authorDisplayName`, `authorAvatarUrl`, `authorGymId` — passing the DTO directly avoids 3 separate params and keeps the call site clean (`PublicProfileHero(view: view)`).
- `StatelessWidget` — no state, no providers. Pure projection of the DTO into pixels.
- The DTO dependency is intentional: this widget is single-purpose (it only renders inside `PublicProfileScreen`'s data branch). If a future screen needs the same hero with different data, refactor extracts a smaller `_HeroBody({avatarUrl, name, gymId})` private widget — mechanical.

### 2.3 `PublicProfileStatsRow`

```dart
class PublicProfileStatsRow extends StatelessWidget {
  const PublicProfileStatsRow({super.key});
}
```

**Rationale**:
- Zero params. All four values are hardcoded `'0'` in this etapa per locked decision 6 (proposal §4). The labels are hardcoded too. No `ConsumerWidget` because there's nothing to watch.
- When Fase 4 wires real stats, the refactor is mechanical: add a `StatsViewModel` param + replace the four `'0'` literals with `model.workouts`, `model.racha`, etc. The widget tree shape does not change.

### 2.4 `PublicProfileFollowButton`

```dart
class PublicProfileFollowButton extends ConsumerWidget {
  const PublicProfileFollowButton({
    super.key,
    required this.friendship,
    required this.viewerUid,
    required this.targetUid,
  });

  final Friendship? friendship;
  final String viewerUid;
  final String targetUid;

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

**Rationale**:
- `friendship` nullable — `null` is the "not following" state (most common for new profile visits).
- `viewerUid` and `targetUid` are required and ALWAYS resolved upstream (the screen only renders this button when `view.isSelf == false`, so both UIDs are guaranteed present).
- `ConsumerWidget` because `request()`/`accept()` taps MUST call `ref.invalidate(friendshipByPairProvider((viewerUid: viewerUid, targetUid: targetUid)))` to refetch state. Without `ref`, the button cannot trigger the invalidation.
- Passing `friendship` directly (instead of reading the provider here) keeps this widget pure and trivially testable in isolation — the parent screen decides what `Friendship?` looks like in the test.

### 2.5 `_ProfileTabPills` (private to `public_profile_screen.dart`)

```dart
class _ProfileTabPills extends ConsumerWidget {
  const _ProfileTabPills({required this.targetUid});

  final String targetUid;

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

**Rationale**:
- Private (`_` prefix) because there is a single consumer — this screen — and the spec REQ-PROFILE-TABS-001 explicitly forbids reuse from `feed_segment_pills.dart`. Keeping it private signals "do not import elsewhere".
- `targetUid` is the family key for `profileTabProvider(targetUid)` (see §4). Without it, two simultaneous profile visits would share tab state — wrong.
- `ConsumerWidget` for the same reason `FeedSegmentPills` is: tap writes `ref.read(profileTabProvider(targetUid).notifier).state = ProfileTab.X`.

### 2.6 No `PublicProfileEmptyState` widget

The tabs' empty state is rendered inline as a small private widget `_ProfileTabBody` inside `public_profile_screen.dart`. Promoting it to a standalone widget would add a file and an import for ~20 LOC of layout. Spec REQ-PROFILE-TABS-002 requires only that the COPY appear correctly — the widget shape is free. We keep it inline to minimize new files.

If Fase 4/5 wires real content into either tab, that work can split `_ProfileTabBody` into two named StatelessWidgets at that time.

## 4. Provider composition chain

All three new providers live in `lib/features/feed/application/public_profile_providers.dart`. The file imports `authStateChangesProvider` from `lib/features/auth/application/auth_providers.dart`, `friendshipRepositoryProvider` from `friendship_providers.dart` (READ-only cross-file dep — we do NOT modify that file), and `postRepositoryProvider` from `post_providers.dart`.

### 4.1 `FriendshipPair` record typedef

Dart 3 records make this a one-liner; no class needed:

```dart
typedef FriendshipPair = ({String viewerUid, String targetUid});
```

The record is structurally equal — two `FriendshipPair` records with the same `viewerUid` and `targetUid` compare equal and hash to the same value, which is exactly what Riverpod's family cache needs. This is preferable to a `class FriendshipPair { ... }` with manual `==` / `hashCode` because (a) less code, (b) impossible to forget equality, (c) idiomatic Dart 3.

### 4.2 `friendshipByPairProvider`

```dart
final friendshipByPairProvider = FutureProvider.autoDispose
    .family<Friendship?, FriendshipPair>((ref, pair) async {
  final viewer = ref.watch(authStateChangesProvider).valueOrNull;
  if (viewer == null) return null;          // auth-gate (SCENARIO-195)
  final repo = ref.watch(friendshipRepositoryProvider);
  return repo.getByPair(pair.viewerUid, pair.targetUid);
});
```

Auth-gate is explicit: when there is no signed-in user, the provider returns `null` WITHOUT calling the repo (SCENARIO-195 asserts the repo mock is NOT invoked). `autoDispose` because the cache key is per-pair and the family would grow unbounded across many profile visits otherwise.

### 4.3 `firstPostByAuthorProvider`

```dart
final firstPostByAuthorProvider =
    FutureProvider.autoDispose.family<Post?, String>((ref, targetUid) async {
  final viewer = ref.watch(authStateChangesProvider).valueOrNull;
  if (viewer == null) return null;          // auth-gate (SCENARIO-198)
  final firestore = ref.watch(firestoreProvider);
  final snap = await firestore
      .collection('posts')
      .where('authorUid', isEqualTo: targetUid)
      .orderBy('createdAt', descending: true)
      .limit(1)
      .get();
  if (snap.docs.isEmpty) return null;       // no-posts fallback (SCENARIO-197)
  return Post.fromJson(snap.docs.first.data());
});
```

The query is exactly: `where('authorUid', isEqualTo: targetUid).orderBy('createdAt', descending: true).limit(1)`. This intentionally bypasses `PostRepository.byAuthor()` because that method does NOT order or limit — and adding those two clauses to the repo method would be a contract change outside our scope. Going to Firestore directly via `firestoreProvider` is the minimal-surface approach. `firestoreProvider` is imported from `lib/features/profile/application/user_providers.dart` (same import path the existing `postRepositoryProvider` uses).

### 4.4 `publicProfileViewProvider` — composition

```dart
final publicProfileViewProvider = FutureProvider.autoDispose
    .family<PublicProfileView, String>((ref, targetUid) async {
  final viewer = ref.watch(authStateChangesProvider).valueOrNull;
  final isSelf = viewer != null && viewer.uid == targetUid;

  final post = await ref.watch(firstPostByAuthorProvider(targetUid).future);

  final Friendship? friendship;
  if (viewer == null || isSelf) {
    friendship = null;                       // skip lookup (SCENARIO-203)
  } else {
    friendship = await ref.watch(friendshipByPairProvider(
      (viewerUid: viewer.uid, targetUid: targetUid),
    ).future);
  }

  // Approach A: denormalized from Post. If target has never posted, falls back
  // to 'Anónimo'. See proposal §4 decision 1.
  return PublicProfileView(
    authorDisplayName: post?.authorDisplayName ?? 'Anónimo',
    authorAvatarUrl: post?.authorAvatarUrl,
    authorGymId: post?.authorGymId,
    friendship: friendship,
    isSelf: isSelf,
  );
});
```

Auth-gate behavior in each provider:
- `friendshipByPairProvider`: returns `null` when `viewer == null` (no Firestore call). Tested by SCENARIO-195.
- `firstPostByAuthorProvider`: returns `null` when `viewer == null` (no Firestore call). Tested by SCENARIO-198.
- `publicProfileViewProvider`: still constructs a valid `PublicProfileView` even when `viewer == null` — `isSelf` resolves to `false`, `friendship` to `null`, and the screen falls back to "Anónimo" if no post is returned. In practice the route is auth-gated by `authRedirect` (router.dart), so unauthenticated callers never reach this provider; the null-handling is defense-in-depth.

The `isSelf` derivation `view.isSelf = (viewer.uid == targetUid)` happens INSIDE this provider. `PublicProfileScreen` reads `view.isSelf` and never recomputes it.

## 5. `PublicProfileView` DTO

```dart
import 'package:freezed_annotation/freezed_annotation.dart';

import 'friendship.dart';

part 'public_profile_view.freezed.dart';

/// Read-only view-model for /feed/profile/:uid. Composed by
/// publicProfileViewProvider — never stored in Firestore (no fromJson/toJson).
///
/// Approach A: author* fields denormalized from the target's most recent Post.
/// If target has never posted, authorDisplayName defaults to 'Anónimo' and the
/// avatar/gym fields are null. See proposal §4 decision 1.
@freezed
class PublicProfileView with _$PublicProfileView {
  const factory PublicProfileView({
    required String authorDisplayName,
    required String? authorAvatarUrl,
    required String? authorGymId,
    required Friendship? friendship,
    required bool isSelf,
  }) = _PublicProfileView;
}
```

Notes:
- Only `part 'public_profile_view.freezed.dart';` — NO `part '*.g.dart';` and NO `factory PublicProfileView.fromJson(...)`. This DTO never crosses the wire (REQ-PROFILE-DTO-001).
- All 5 fields are `required`. Freezed generates structural `==` / `hashCode` covering all 5 (SCENARIO-199, SCENARIO-200).
- `authorDisplayName` is non-nullable because the provider always supplies a fallback (`'Anónimo'`). The DTO itself does NOT carry a default — the contract for the default lives in the provider, where the source data (`post`) is also visible.

Build step: after writing the file, run `dart run build_runner build --delete-conflicting-outputs` (Strict TDD: only after the DTO test is committed RED, then build, then test passes GREEN).

## 6. `FriendshipRepository.getByPair` signature

```dart
// lib/features/feed/data/friendship_repository.dart (additive — keep all existing methods unchanged)

/// Returns the friendship doc between [uidA] and [uidB] if it exists,
/// or null if no doc exists at the deterministic sortedDocId.
///
/// Reads the single doc at `friendships/{sortedDocId(uidA, uidB)}` — no
/// `where()` clause, no collection scan. Argument order is irrelevant because
/// the doc ID is sorted lexicographically (SCENARIO-192).
///
/// No new Firestore rules required: the viewer is always a `member` of any
/// doc that exists between (uidA, uidB), and a non-existent doc returns null
/// from `.get()` without a permission error.
Future<Friendship?> getByPair(String uidA, String uidB) async {
  final id = Friendship.sortedDocId(uidA, uidB);
  final snap = await _friendships.doc(id).get();
  return _fromDoc(snap);   // _fromDoc returns null when !snap.exists
}
```

Rationale for using `_fromDoc(snap)` (existing private helper) instead of inlining the `if (!snap.exists) return null;` check: consistency with `request()` and the existing patterns in this repo. `_fromDoc` already handles the `!snap.exists || data == null` case correctly.

The spec sketch in the launch prompt uses `Friendship.fromJson({...doc.data()!, 'id': doc.id})` — but the existing repo's `_fromDoc` already strips that responsibility, so we DON'T duplicate. The behavior is identical (returns null when no doc; returns a fully hydrated `Friendship` otherwise).

## 7. 4-state SEGUIR button state machine

State resolution lives inline in `public_profile_follow_button.dart`. No separate `enum FollowState` is needed — the `switch` is exhaustive and clearer than enumerating intermediate states. But for documentation purposes:

```dart
// Conceptual state — not declared in code; the switch below is exhaustive.
enum _FollowState { notFollowing, requestSent, requestReceived, following }

_FollowState _resolveState(Friendship? friendship, String viewerUid) {
  if (friendship == null) return _FollowState.notFollowing;
  if (friendship.status == FriendshipStatus.accepted) return _FollowState.following;
  if (friendship.requesterId == viewerUid) return _FollowState.requestSent;
  return _FollowState.requestReceived;   // pending && requesterId == targetUid
}
```

Actual implementation in `build`:

```dart
Widget build(BuildContext context, WidgetRef ref) {
  final palette = AppPalette.of(context);
  final repo = ref.watch(friendshipRepositoryProvider);

  Future<void> invalidate() async => ref.invalidate(
        friendshipByPairProvider((viewerUid: viewerUid, targetUid: targetUid)),
      );

  // 4-state resolution — must match REQ-PROFILE-FOLLOW-001 truth table.
  if (friendship == null) {
    return _FollowPill(
      label: 'SEGUIR',
      style: _FollowPillStyle.mintFilled,
      onTap: () async {
        await repo.request(viewerUid, targetUid);
        invalidate();
      },
    );
  }
  if (friendship!.status == FriendshipStatus.accepted) {
    return _FollowPill(
      label: 'SIGUIENDO',
      style: _FollowPillStyle.outlined,
      leadingIcon: TreinoIcon.check,
      onTap: null,                          // unfollow deferred to Fase 5
    );
  }
  if (friendship!.requesterId == viewerUid) {
    return _FollowPill(
      label: 'SOLICITUD ENVIADA',
      style: _FollowPillStyle.outlinedMuted, // 0.6 opacity wrapper
      onTap: null,
    );
  }
  // friendship pending && requesterId == targetUid → received → ACEPTAR
  return _FollowPill(
    label: 'ACEPTAR',
    style: _FollowPillStyle.mintFilled,
    onTap: () async {
      await repo.accept(friendship!.id, viewerUid);
      invalidate();
    },
  );
}
```

Visual mapping per state:

| State | Label | Background | Border | Text color | Leading icon | onTap |
|---|---|---|---|---|---|---|
| `notFollowing` (`friendship == null`) | `'SEGUIR'` | `palette.accent` (mint fill) | `palette.accent` | `palette.bg` | none | `repo.request(viewerUid, targetUid)` then `invalidate()` |
| `requestSent` (pending, viewer is requester) | `'SOLICITUD ENVIADA'` | transparent | `palette.border` | `palette.textMuted` (wrapped in `Opacity(0.6)`) | none | `null` (no-op) |
| `requestReceived` (pending, target is requester) | `'ACEPTAR'` | `palette.accent` (mint fill) | `palette.accent` | `palette.bg` | none | `repo.accept(friendship.id, viewerUid)` then `invalidate()` |
| `following` (`status == accepted`) | `'SIGUIENDO'` | transparent | `palette.border` | `palette.textPrimary` | `TreinoIcon.check` (size 14, `palette.textPrimary`) | `null` (no-op; unfollow Fase 5) |

`_FollowPill` is a private widget (one shape, parameterized by style + label + icon + tap). Shape is `BorderRadius.circular(20)` + padding `horizontal: 18, vertical: 12`. The `outlinedMuted` style wraps the pill in `Opacity(opacity: 0.6)` per REQ-PROFILE-FOLLOW-001 (SCENARIO-221 asserts the opacity widget is in the tree).

MENSAJE stub button (`_MessageButtonStub` private in the screen file): same `_FollowPill` shape, label `'MENSAJE'`, style `outlinedMuted`, `onTap: null`. Source comment: `// Stub: wired in Fase 5 (Coach chat).`

Re Accept optimistic update: we DO NOT optimize. The flow is `await repo.accept(...)` → `ref.invalidate(...)` → provider refetches → UI rebuilds with `friendship.status == accepted`. This is simpler than optimistic state, has no inconsistency window, and matches the existing pattern from `feed-shell-amigos`. Cost: one extra Firestore read per accept tap. Acceptable at current scale.

## 9. Edge case decisions

### 9.1 Gym name lookup

`gymNameFromId(String? gymId) → String` lives in `lib/features/feed/domain/gym_name.dart`. It uses a `const` map mirroring the 3 entries in `profile_setup_providers.dart` `_kHardcodedGyms`:

```dart
import '../../profile_setup/domain/gym.dart' show kNoGymId;

/// Resolves a gymId to the display name from the hardcoded catalog.
///
/// Mirrors the catalog in profile_setup_providers.dart `_kHardcodedGyms`.
/// When the gyms/ Firestore collection is introduced (Fase 4+), replace this
/// with a Riverpod provider that hydrates from Firestore.
const Map<String, String> _kGymNames = {
  'smart-fit-palermo': 'Smart Fit Palermo',
  'sportclub-belgrano': 'SportClub Belgrano',
  'megatlon-recoleta': 'Megatlon Recoleta',
};

String gymNameFromId(String? gymId) {
  if (gymId == null || gymId.isEmpty || gymId == kNoGymId) return '';
  return _kGymNames[gymId] ?? gymId.toUpperCase();
}
```

Rules:
- `null` → `''` (hide subtitle)
- `''` → `''` (defensive)
- `'no-gym'` (the `kNoGymId` sentinel) → `''`
- known id → resolved display name (e.g. `'megatlon-recoleta'` → `'Megatlon Recoleta'`)
- unknown id → `gymId.toUpperCase()` (best-effort fallback; visible signal that the catalog is stale)

We do NOT import `_kHardcodedGyms` from `profile_setup_providers.dart` because that constant is private. Instead we maintain a small parallel `_kGymNames` map here — the duplication is intentional and trivial (3 entries) until Firestore replaces both. We DO import `kNoGymId` from `gym.dart` because it is public (and the canonical source of the sentinel string).

### 9.2 No-posts fallback

When `firstPostByAuthorProvider(targetUid)` returns `null`:
- `view.authorDisplayName == 'Anónimo'` (default applied by the provider)
- `view.authorAvatarUrl == null`
- `view.authorGymId == null`

The screen still renders normally:
- `PublicProfileHero` shows `'ANÓNIMO'` (uppercase) + no gym subtitle + `PostAvatar` with `'?'` initial (because `_computeInitial('Anónimo') == '?'` per the existing helper in `post_avatar.dart` line 43-46).
- Stats row, tabs, follow button all render regardless of post existence.

This is the explicit Approach A trade-off (locked decision 1, propose §4).

### 9.3 Self-visit

When `view.isSelf == true`:
- Follow button + MENSAJE row are entirely OMITTED from the widget tree (REQ-PROFILE-SCREEN-002, SCENARIO-208). Not just disabled — absent.
- The trailing `SizedBox(height: 20)` after the row is also omitted (vertical rhythm stays tight).
- Hero, stats row, tab pills, and tab bodies all render normally (SCENARIO-209).
- `view.friendship` is `null` (skipped lookup per the provider composition rules §4.4). The follow button's state-machine logic is irrelevant because the widget is not in the tree.

### 9.4 Accept button — NOT optimistic

When user taps `'ACEPTAR'`:
1. `await repo.accept(friendship.id, viewerUid)` fires.
2. On success: `ref.invalidate(friendshipByPairProvider((viewerUid, targetUid)))` — the provider refetches.
3. `publicProfileViewProvider` rebuilds (it `ref.watch`es `friendshipByPairProvider`) → new `PublicProfileView` with `friendship.status == accepted`.
4. UI rebuilds → button switches to `'SIGUIENDO'` + check icon.

If `repo.accept(...)` throws (e.g. permission error), the await rethrows in the gesture handler. We do NOT catch — Flutter's default error handler shows the exception in dev; in prod the tap is a no-op (the button stays in `'ACEPTAR'` state because no invalidation happened). This is the same posture as `feed-shell-amigos`'s post creation. Adding explicit error UI is out of scope for this etapa.

Cost: one extra Firestore read per accept tap (provider refetch). Acceptable at current scale (single user action, low-traffic product).

### 9.6 MENSAJE button

### 9.7 Tabs default

`profileTabProvider(targetUid)` initial state is `ProfileTab.rutinas`. Provider declaration:

```dart
final profileTabProvider =
    StateProvider.autoDispose.family<ProfileTab, String>(
  (ref, _) => ProfileTab.rutinas,
);
```

This is a private file-scope provider (inside `public_profile_screen.dart`). `autoDispose` ensures the tab state resets when the user leaves and returns to the profile. The family key is `targetUid` so two simultaneous profile visits (e.g. push from profile A's hero to profile B, then back) do not share tab state.

`ProfileTab` enum lives in `lib/features/feed/domain/profile_tab.dart` (NOT inside `public_profile_screen.dart`, because Riverpod's family key must be a stable value across the screen file — keeping the enum in a domain file avoids any cyclic import surprises):

```dart
enum ProfileTab { rutinas, actividad }
```

The enum is intentionally NOT exported through a barrel — only `public_profile_screen.dart` imports it. Other features have no reason to depend on this type.
