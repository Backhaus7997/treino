## 2. Widget API surface

All widgets live in the `home` feature. No widget reads providers — `HomeScreen` is the only `ConsumerWidget`. Data flow is one-way down via constructor params.

### 2.1 `HomeScreen`

```dart
class HomeScreen extends ConsumerWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

Rationale: zero parameters. Owns the single `ref.watch(userProfileProvider)` call (REQ-HOME-SCREEN-001). Keeps the public route handle in `router.dart` unchanged.

### 2.2 `HomeHeader`

```dart
class HomeHeader extends StatelessWidget {
  const HomeHeader({
    super.key,
    required this.profile,
  });

  final UserProfile? profile;
}
```

Rationale: takes the nullable profile by parameter so the screen owns the `.when(...)` reduction. `null` is a valid value and triggers the fallback branch (REQ-HOME-HEADER-002, 004). `UserProfile?` is the only public type dependency — import path `package:treino/features/profile/domain/user_profile.dart`.

### 2.3 `EmpezarEntrenamientoCard`

```dart
class EmpezarEntrenamientoCard extends StatelessWidget {
  const EmpezarEntrenamientoCard({super.key});
}
```

Rationale: zero parameters by spec (REQ-HOME-EMPEZAR-001). All literal strings live inside this file as private `static const` so Etapa 5 can swap to provider-driven data without renaming the widget.

### 2.4 `EstaSemanaCard`

```dart
class EstaSemanaCard extends StatelessWidget {
  const EstaSemanaCard({super.key});
}
```

Rationale: zero parameters by spec (REQ-HOME-SEMANA-001). Placeholder content lives inline; Etapa 5 replaces the body, not the wrapper.

### 2.5 `HomeCTAButton`

```dart
class HomeCTAButton extends StatelessWidget {
  const HomeCTAButton({
    super.key,
    required this.label,
    this.onPressed,
    this.leadingIcon,
  });

  final String label;
  final VoidCallback? onPressed;
  final IconData? leadingIcon;
}
```

Rationale:
- `label` required — every button must declare its purpose (REQ-HOME-CTA-001).
- `onPressed` nullable so the card can pass `null` until Etapa 5 wires navigation (REQ-HOME-EMPEZAR-004, REQ-HOME-CTA-003).
- `leadingIcon` nullable `IconData?` — accepts `TreinoIcon.play` etc., null means no icon (REQ-HOME-CTA-005). Using `IconData?` (not a `Widget?` slot) keeps the API minimal and matches `AuthPillButton`'s established style.
- No `isLoading`. `AuthPillButton` has it because Auth shows pending network state; Home CTA has no network at Etapa 1. Adding it now would be YAGNI; the future "Empezar" wire in Etapa 5 can add it then.

## 4. State flow

### 4.1 Single watch point

`HomeScreen.build` is the ONLY place that touches `userProfileProvider`. Pattern (normative shape):

```dart
@override
Widget build(BuildContext context, WidgetRef ref) {
  final palette = AppPalette.of(context);
  final profileAsync = ref.watch(userProfileProvider);

  final Widget headerOrSkeleton = profileAsync.when(
    data: (profile) => HomeHeader(profile: profile),       // profile may be null
    loading: () => const _HomeHeaderSkeleton(),
    error: (_, __) => const HomeHeader(profile: null),     // silent fallback
  );

  return Padding(
    padding: const EdgeInsets.symmetric(horizontal: 20),
    child: ListView(
      padding: const EdgeInsets.symmetric(vertical: 20),
      physics: const ClampingScrollPhysics(),
      children: [
        headerOrSkeleton,
        const SizedBox(height: 20),
        const EmpezarEntrenamientoCard(),
        const SizedBox(height: 12),
        const EstaSemanaCard(),
      ],
    ),
  );
}
```

Key invariants:

| Async state | Header slot | Cards |
|---|---|---|
| `AsyncData(profile)` (profile != null) | `HomeHeader(profile: profile)` — real name + avatar | rendered |
| `AsyncData(null)` | `HomeHeader(profile: null)` — generic "HOLA!" + initials "?" | rendered |
| `AsyncLoading()` | `_HomeHeaderSkeleton()` — 56px fixed-height box | rendered |
| `AsyncError(_, _)` | `HomeHeader(profile: null)` — same as data(null), silent fallback | rendered |

Cards are rendered in all four branches because they are hardcoded — they don't depend on profile data. REQ-HOME-PROVIDER-003 scenario 2 confirms `EmpezarEntrenamientoCard` and `EstaSemanaCard` are findable during loading.

### 4.2 `UserProfile?` destructuring (inside `HomeHeader.build`)

```dart
final profile = this.profile;                                  // local capture
final displayName = profile?.displayName;                      // String?
final avatarUrl = profile?.avatarUrl;                          // String?

final greetingText = (displayName != null && displayName.isNotEmpty)
    ? 'HOLA, ${displayName.toUpperCase()}!'
    : 'HOLA!';

final initials = (displayName != null && displayName.isNotEmpty)
    ? displayName.characters.first.toUpperCase()
    : '?';
```

Notes:
- `.characters.first` (from `dart:core` `String.characters` via `package:characters`) handles multi-code-point graphemes safely. For ASCII like "Martín" → "M". This package is part of Flutter's transitive deps; no new dep needed.
- See §9 decision 2 for the full rationale on initials (single letter, not "MB" from "Martín Backhaus").
- `displayName.toUpperCase()` is enough — no need for `TextCapitalization` because we control the literal. Dart's `toUpperCase()` uses Unicode rules and handles "ñ", "í" correctly.

### 4.3 Why `_HomeHeaderSkeleton` is private to `home_screen.dart` (not to `home_header.dart`)

The skeleton is presentational glue between the loading branch of `userProfileProvider` and the layout. Putting it inside `HomeHeader` would force `HomeHeader` to expose a loading mode and read state, contradicting REQ-HOME-SCREEN-001 ("no child widget may call `ref.watch`"). Keeping the skeleton private to the screen file keeps `HomeHeader` strictly state-free.

## 6. Layout / spacing decisions

Allowed set per AGENTS.md: `{8, 12, 14, 18, 20}`. Every gap, padding, and margin in this PR maps to one of these.

| Position | Value | Reason |
|---|---|---|
| `HomeScreen` outer horizontal padding | `20` | Matches the standard screen-edge padding used in `profile_setup` flows. |
| `HomeScreen` `ListView` vertical padding | `20` | Symmetric top/bottom inside the SafeArea. |
| Gap: header → first card | `20` | Largest gap in the screen, separates greeting block from content blocks. |
| Gap: between cards | `12` | Mockup reads ~16px; allowed set has no 16, and 14 reads too tight against `r-lg=20` rounded corners. **12 is the chosen compromise — see §9 decision 4.** |
| Card outer padding (inside `Container`) | `18` | Standard card breathing room; matches `ExperienceCard` precedent (`EdgeInsets.all(18)`). |
| Card r-lg | `20` | `r-lg` per design-system tokens. |
| Card border width | `1` | Per design-system. |
| `EmpezarEntrenamientoCard` internal gap: label → hero | `8` | Tight visual relationship between label and hero. |
| `EmpezarEntrenamientoCard` internal gap: hero → subtitle | `8` | Same tight pairing. |
| `EmpezarEntrenamientoCard` internal gap: subtitle → stat row | `14` | Slight breathing before stats. |
| `EmpezarEntrenamientoCard` internal gap: stat row → CTA | `18` | Larger breathing room before the CTA pill. |
| Stat row inter-icon-text gap | `8` | Icon-to-text adjacency. |
| Stat row inter-stat gap | `18` | Between "6 ejercicios" and "~55 min". |
| `EstaSemanaCard` internal gap: title → body | `12` | Single readable spacing. |
| `HomeCTAButton` height | `56` | Fixed pill height; matches `AuthPillButton` (acceptable since 56 is not a "gap" — it's a control size). |
| `HomeCTAButton` icon → text gap | `8` | Adjacency. |
| Avatar circle size | `56 × 56` | Control size, not a gap. Mockup-derived. |
| Avatar → greeting gap | `12` | Adequate separation in the header row. |

The two non-allowed-set values (`56` for button height, `56×56` for avatar) are **control sizes**, not gaps/paddings. They're permitted because the allowed-set rule governs spacing tokens, not widget dimensions. Both match precedent (`AuthPillButton` height = 56).

### 8.2 Provider overrides

Only `home_screen_test.dart` needs overrides. Pattern:

```dart
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:treino/features/profile/application/user_providers.dart';

// AsyncData(profile)
userProfileProvider.overrideWith(
  (ref) => Stream.value(makeProfile(displayName: 'Martín', avatarUrl: 'https://...')),
)

// AsyncData(null)
userProfileProvider.overrideWith((ref) => Stream<UserProfile?>.value(null))

// AsyncLoading
userProfileProvider.overrideWith((ref) => const Stream<UserProfile?>.empty())

// AsyncError
userProfileProvider.overrideWith((ref) => Stream<UserProfile?>.error(Exception('network')))
```

The provider is a `StreamProvider<UserProfile?>`, so overrides return a `Stream<UserProfile?>`. `Stream.empty()` resolves to `AsyncLoading` because no event is ever emitted. `Stream.error(...)` resolves to `AsyncError`.

After each override, the test MUST call `await tester.pump(const Duration(milliseconds: 50))` (or `pumpAndSettle`) before asserting, except for the loading branch where a single `pump()` (no settle) keeps the provider in `AsyncLoading`.

## 9. Edge cases and decisions

### Decision 1 — Loading state visual: shimmer-only-on-header

**Choice**: only the header has a placeholder; the two cards render normally during `AsyncLoading`.

**Why**: cards are hardcoded (no profile data dependency). Hiding them during a transient `userProfileProvider` loading state would create unnecessary flashes. The header is the only profile-dependent element, so its skeleton sits in for it until `AsyncData` arrives. A real `shimmer` package would add a dependency for visual polish that isn't worth it at Etapa 1 — a flat-color `SizedBox(height: 56)` is enough to prevent layout jump (REQ-HOME-PROVIDER-003).

**Implementation**: `_HomeHeaderSkeleton` is a private widget inside `home_screen.dart` returning a `SizedBox(height: 56)` (matches the 56×56 avatar height). No animation, no gradient. If a future PR wants a real shimmer, it replaces this one private widget.

**Rejected alternatives**:
- Whole-screen skeleton: too much movement, looks broken when the cards never change.
- Shimmer package: extra dep, premature polish.

### Decision 2 — Initials fallback logic

**Choice**: `"Martín"` → `"M"`. `"Martín Backhaus"` → `"M"`. Empty / null → `"?"`.

**Why**: the avatar circle is 56×56. With `Barlow Condensed w700` at 22px, a single character fits comfortably in the circle. Two characters ("MB") would either need to shrink the font size further or risk clipping. Most fitness/social apps default to single-initial avatars. The propose document risk #1 mentions `"?" or "TR"` — picking `"?"` keeps the fallback intent (clearly unknown) without inventing the "TR" brand initials at runtime.

**Algorithm**:

```dart
String _computeInitials(String? displayName) {
  if (displayName == null || displayName.isEmpty) return '?';
  return displayName.characters.first.toUpperCase();
}
```

**Rejected alternatives**:
- "TB" from "Tincho Backhaus": requires splitting on whitespace, choosing between first+last vs first+second, and handling single-word names. Adds branches with no product value at this stage.
- Empty string: leaves an empty circle that reads as a broken render.
- "TR" (brand initials): co-opts brand identity into a user fallback. Confusing.

### Decision 3 — Disabled CTA appearance

**Choice**: 0.5 alpha on the background color when `onPressed == null`. Foreground (text) stays at full opacity.

**Why**: matches the established `AuthPillButton` pattern (`disabledBackgroundColor: palette.accent.withValues(alpha: 0.5)`). Avoids inventing a second visual language for disabled CTAs. Visually communicates "not active right now" without making the label unreadable.

**Implementation**:

```dart
ElevatedButton(
  onPressed: onPressed,                              // pass through; null disables
  style: ElevatedButton.styleFrom(
    backgroundColor: palette.accent,
    foregroundColor: palette.bg,
    disabledBackgroundColor: palette.accent.withValues(alpha: 0.5),
    disabledForegroundColor: palette.bg,
    shape: const StadiumBorder(),
    padding: EdgeInsets.zero,
  ),
  // ...
)
```

`ElevatedButton` natively switches between `backgroundColor`/`disabledBackgroundColor` based on whether `onPressed` is null — no manual ternary needed.

**Rejected alternatives**:
- Whole-button `Opacity(0.5)`: dims the text too, harder to read.
- Different color (gray): introduces a non-palette color. Banned by AGENTS.md.

### Decision 4 — Inter-card gap of 12 (not 16)

**Choice**: `SizedBox(height: 12)` between `EmpezarEntrenamientoCard` and `EstaSemanaCard`.

**Why**: the allowed set is `{8, 12, 14, 18, 20}` — there is no 16. The mockup reads around 16px. `12` keeps the cards visually grouped without crowding; `14` reads tight against `r-lg=20` rounded corners; `18` reads too detached. This is a hard constraint, not a preference — call it out so the apply implementer doesn't try to "fix" it with 16.

**Rejected alternatives**: anything outside the allowed set, including 16.
