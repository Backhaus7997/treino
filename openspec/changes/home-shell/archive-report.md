## Deliverables Summary

### What was built

A complete home screen shell for the TREINO fitness app (Fase 2 · Etapa 1):

- **HomeScreen** (`ConsumerWidget`): Orchestrator widget watching `userProfileProvider` with 4 async states (data, loading, error, null)
- **HomeHeader**: Greeting ("HOLA, {displayName}!" or "HOLA!") + circular avatar (via `CachedNetworkImage` or gradient fallback with initials)
- **EmpezarEntrenamientoCard**: Hardcoded workout card (day/date, workout name "PUSH", muscle groups, exercise count, duration, disabled CTA pill)
- **EstaSemanaCard**: Placeholder card for weekly stats (no streak, no SVG, no muscle map — those are Etapa 5)
- **HomeCTAButton**: Reusable pill CTA widget (accent background, StadiumBorder, optional leading icon; can be promoted to core later)

### Architecture decisions

- **Single ConsumerWidget at screen level** — all child widgets are stateless; `HomeScreen` owns the single `ref.watch(userProfileProvider)` call
- **Component composition over monolith** — mirrors `profile_setup` precedent; each widget independently testable
- **Hardcoded content at Etapa 1** — all card content is `static const` strings, ready for Etapa 5 provider-driven replacement
- **No routing wires, no domain logic** — placeholder `onPressed: null` on CTA waits for Etapa 5 session navigation
- **Strict TDD order** — all tests written before widgets; 247 tests green at archive time

## Discoveries & Lessons Learned

### 1. Unicode play glyph on Windows renders as colored emoji

**Issue**: `EmpezarEntrenamientoCard` initially had `label: '▶ EMPEZAR ENTRENAMIENTO'` (U+25B6). On Windows, the system renders this via Segoe UI Emoji (colored emoji font) producing an orange/red square instead of a clean monochrome triangle.

**Discovery**: Automated tests did NOT catch this because widget tests use Material's text rendering stack, which is platform-agnostic. Smoke testing on a real Windows device caught it.

**Resolution**: Moved the glyph to `leadingIcon: TreinoIcon.play` parameter instead of baking it into the label string. `TreinoIcon` is a custom icon font ensuring consistent render across platforms.

**Lesson**: Smoke testing on actual target devices is irreplaceable — platform-specific emoji/font rendering edge cases won't be caught by CI. Test suite was comprehensive but orthogonal to this visual defect.

**Apply-phase decision**: After the fix, post-verify identified that `spec.md`, `tasks.md`, and `design.md` still referenced the old label string. Orchestrator updated all 3 artifact files to sync with corrected implementation.

### 2. Placeholder content strategy works well

The hardcoded content (`static const` strings) in `EmpezarEntrenamientoCard` and `EstaSemanaCard` made it trivial for Etapa 5 to swap in real provider data — the widget signatures don't change, only the body logic. This confirms Approach B was the right call.

### 3. Single-watch architecture is maintainable

Centralizing the `userProfileProvider` watch at the screen level and passing data down as constructor parameters kept the test surface clean (no provider overrides needed in child tests). The `.when(data: loading: error:)` pattern at one point eliminated branching complexity.

### 4. Strict TDD order prevented rework

Writing tests first (RED) for leaf widgets (`HomeCTAButton`, `EstaSemanaCard`) before composites (`EmpezarEntrenamientoCard`, `HomeHeader`, `HomeScreen`) meant no unimplemented dependencies. Each commit was green from day one — zero "I'll fix the tests later" debt.

### 5. Design document precision caught ambiguities

The design's §9 (edge cases and decisions) and §10 (implementation order) were specific enough that the apply implementer had zero ambiguity on:
- Allowed spacing values (`{8, 12, 14, 18, 20}` — no 16, no 24)
- Initials logic (single char + `"?"` fallback, not "MB")
- Card gap (12, not 14 or 16)
- Disabled CTA appearance (0.5 alpha, not gray)

This level of precision is critical for independent implementers to produce nearly-identical code.
