## 2. Widget API surfaces

All widgets live in the `feed` feature. `FeedScreen` is the only `ConsumerWidget`. All children receive their data via constructor params (one-way down). The exception is `FeedSegmentPills`, which is a `ConsumerWidget` because it needs to read+write `feedSegmentProvider` directly — promoting that state up to `FeedScreen` would force the screen to rebuild on every tap.

### 2.1 `FeedScreen`

```dart
class FeedScreen extends ConsumerWidget {
  const FeedScreen({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

**Rationale**: zero parameters. Owns the single `ref.watch(feedSegmentProvider)` and `ref.watch(myFriendsFeedProvider)` calls. Keeps the public route handle in `router.dart` unchanged.

### 2.2 `_FeedHeader` (private to `feed_screen.dart`)

```dart
class _FeedHeader extends StatelessWidget {
  const _FeedHeader();
}
```

**Rationale**: header is a layout-only `Row` ("FEED" + search icon + plus icon). Private + parameterless because all content is hardcoded for Etapa 2 and the icon taps are explicit stubs (Etapa 5 wires them). Lives inline in the screen file — promoting it to its own file would invite premature reuse.

### 2.3 `FeedSegmentPills`

```dart
class FeedSegmentPills extends ConsumerWidget {
  const FeedSegmentPills({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref);
}
```

**Rationale**: zero parameters. Reads `feedSegmentProvider` to know which pill is active and writes it on AMIGOS tap. Lives in its own file so `FeedScreen` can `find.byType(FeedSegmentPills)` in tests (REQ-FEED-SCREEN-001, SCENARIO-146) and so the widget owns its own visual contract.

### 2.4 `PostCard`

```dart
class PostCard extends StatelessWidget {
  const PostCard({
    super.key,
    required this.post,
    this.onAuthorTap,
  });

  final Post post;
  final VoidCallback? onAuthorTap;
}
```

**Rationale**:
- `post` is required — every card must declare its source data.
- `onAuthorTap` is nullable (REQ-FEED-POSTCARD-007). Default `null` in Etapa 2 (`FeedScreen` passes nothing). The avatar + name region wraps in a `GestureDetector(onTap: onAuthorTap)` — when null, the gesture detector still mounts but the tap is a no-op (same shape as a regular Flutter button with `onPressed: null`).
- No `onTagTap` parameter — chip navigation is internal (`context.push(...)` from inside the private `_RoutineTagChip`). Spec REQ-FEED-POSTCARD-003 binds the destination, not the indirection.
- `Post` is the only public type dependency. Import: `package:treino/features/feed/domain/post.dart`.

### 2.5 `_RoutineTagChip` (private to `post_card.dart`)

```dart
class _RoutineTagChip extends StatelessWidget {
  const _RoutineTagChip({required this.tag});

  final RoutineTag tag;
}
```

**Rationale**: private because no other feed widget renders a routine chip in Etapa 2. The chip owns its own navigation: `GestureDetector(onTap: () => context.push('/workout/routine/${tag.routineId}'))`. Promoting this to a public widget would invite premature generalisation — current scope has a single consumer.

If a future PR needs to render the chip elsewhere (e.g. workout history), the refactor is mechanical: rename the private class and lift it to `lib/features/feed/presentation/widgets/routine_tag_chip.dart` with its existing API.

### 2.6 `PostAvatar`

```dart
class PostAvatar extends StatelessWidget {
  const PostAvatar({
    super.key,
    required this.authorDisplayName,
    required this.authorAvatarUrl,
    this.size = 40,
  });

  final String authorDisplayName;
  final String? authorAvatarUrl;
  final double size;
}
```

**Rationale**:
- Both display name and avatar URL are required as named params; `authorAvatarUrl` is nullable per the model (REQ-FEED-POST-001).
- `size` defaults to `40` (the value the mockup uses inside `PostCard`). Exposing it as a param allows reuse if a future profile screen wants a larger variant (96px) — no breaking change.
- No `onTap` here. Tap behaviour belongs to the parent (`PostCard` wraps the avatar in its own `GestureDetector` so the entire author region — avatar + name + gym/timestamp — is one tap target).

### 2.7 `FeedEmptyState`

```dart
class FeedEmptyState extends StatelessWidget {
  const FeedEmptyState({super.key});
}
```

**Rationale**: zero parameters by spec (REQ-FEED-EMPTY-001). Copy is hardcoded inside this file as a `static const _kCopy = 'Aún no hay posts de tus amigos';`. Etapa 3+ can swap the copy by lifting it to a parameter when a second consumer materializes.

## 3. Widget tree composition

Notation: `[shell]` = produced by `_ShellScaffold` in `router.dart` (already present, do NOT re-add). `[screen]` = produced inside `FeedScreen.build`. Spacing values are all from the allowed set `{8, 12, 14, 18, 20}` (see §6).

### 3.1 `FeedScreen` tree

```
[shell] Scaffold
[shell]   body: AppBackground
[shell]     SafeArea
[screen]       Padding(horizontal: 20)              // outer screen padding
[screen]         Column(crossAxisAlignment: stretch)
[screen]           SizedBox(height: 20)              // top breathing room inside SafeArea
[screen]           _FeedHeader()                     // see §3.2
[screen]           SizedBox(height: 14)              // header → pills (allowed-set; 16 not allowed)
[screen]           FeedSegmentPills()                // see §3.3
[screen]           SizedBox(height: 14)              // pills → body
[screen]           Expanded(
[screen]             child: switch (segment) {
[screen]               FeedSegment.amigos => _AmigosBody(),     // see §3.4
[screen]               FeedSegment.gym    => SizedBox.shrink(), // REQ-FEED-SCREEN-001 SCENARIO-148
[screen]               FeedSegment.public => SizedBox.shrink(), // REQ-FEED-SCREEN-001 SCENARIO-149
[screen]             },
[screen]           )
```

Key invariants:
- `FeedScreen` introduces **no** `Scaffold`, `AppBackground`, or `SafeArea` (REQ-FEED-SCREEN-001, SCENARIO-147). The outer `Column` lives directly inside the shell-applied `SafeArea`.
- `Expanded` ensures the body fills the remaining viewport so the AMIGOS `ListView` can scroll independently of the header/pills.

### 3.2 `_FeedHeader` tree

```
_FeedHeader
  Row(crossAxisAlignment: center, mainAxisAlignment: spaceBetween)
    Text('FEED')                                                 // Barlow Condensed 700 28pt
    Row(mainAxisSize: min)
      IconButton(icon: TreinoIcon.search,  onPressed: null)      // stub for Etapa 5
      SizedBox(width: 8)                                         // adjacency gap
      IconButton(icon: TreinoIcon.plus,    onPressed: null)      // stub for Etapa 5
```

`IconButton` with `onPressed: null` natively renders as visually disabled; per spec (REQ-FEED-SCREEN-001 SCENARIO-145) the icons must be **findable** (`find.byIcon`) but tap is a no-op. Using `IconButton` (not `GestureDetector`) keeps the touch target at the standard 48×48 — accessibility benefit.

### 3.3 `FeedSegmentPills` tree

```

    Row(mainAxisSize: min)

        label: 'AMIGOS',

      SizedBox(width: 8)
```

`_Pill` is a private widget inside `feed_segment_pills.dart` (copied from `LevelFilterPills._Pill`, **not** imported — Feed owns its own pill visual per ADR-FS-3 below). `_Pill.onTap` is `VoidCallback?` (nullable) so MI GYM and PÚBLICO can pass `null` cleanly; the `GestureDetector` inside `_Pill` becomes a no-op when its callback is null.

### 3.4 `_AmigosBody` tree (private inside `feed_screen.dart`)

```
_AmigosBody  (private to feed_screen.dart, ConsumerWidget)
  ref.watch(myFriendsFeedProvider).when(
    data: (posts) => posts.isEmpty
        ? const FeedEmptyState()
        : ListView.separated(
            physics: ClampingScrollPhysics(),
            padding: EdgeInsets.only(bottom: 20),                // breathing room above bottom bar
            itemCount: posts.length,
            separatorBuilder: (_, __) => const SizedBox(height: 12),
            itemBuilder: (_, i) => PostCard(post: posts[i]),
          ),
    loading: () => Center(
      child: CircularProgressIndicator(color: palette.accent),
    ),
    error: (_, __) => Center(
      child: Text(
        'No pudimos cargar tu feed. Intentá de nuevo.',
        style: TextStyle(color: palette.textMuted),
        textAlign: TextAlign.center,
      ),
    ),
  )
```

`_AmigosBody` is a `ConsumerWidget` (private), not a `ref.watch` inside `FeedScreen.build`. Why: when the segment changes from gym/public back to amigos, isolating the friends watch inside its own widget avoids rebuilding the whole screen on every list change. It's also a clean attachment point for the AsyncValue `.when` reduction.

## 4. State flow / provider composition

### 4.1 Provider topology

```
authStateChangesProvider (StreamProvider<User?>) ─┐
                                                  │
acceptedFriendsProvider(uid) (FutureProvider.family<List<String>, String>) ─┐
                                                                            │
feedForFriendsProvider(uids) (FutureProvider.family<List<Post>, List<String>>) ─┐
                                                                                │
                                          ┌─────────────────────────────────────┘
                                          ▼
                              myFriendsFeedProvider (FutureProvider<List<Post>>)  ◀── single UI entry point
                                          ▲
                                          │
                                  ref.watch(myFriendsFeedProvider)
                                          │
                                       _AmigosBody

feedSegmentProvider (StateProvider<FeedSegment>) ◀── FeedScreen reads / FeedSegmentPills reads+writes
```

### 4.2 `myFriendsFeedProvider` — exact composition

File: `lib/features/feed/application/feed_screen_providers.dart`.

```dart
import 'package:flutter_riverpod/flutter_riverpod.dart';

import '../../auth/application/auth_providers.dart';
import '../domain/feed_segment.dart';
import '../domain/post.dart';
import 'friendship_providers.dart';
import 'post_providers.dart';

final feedSegmentProvider = StateProvider<FeedSegment>(
  (ref) => FeedSegment.amigos,
);

final myFriendsFeedProvider = FutureProvider<List<Post>>((ref) async {
  final auth = await ref.watch(authStateChangesProvider.future);
  if (auth == null) return const <Post>[];

  final friendUids = await ref.watch(acceptedFriendsProvider(auth.uid).future);
  if (friendUids.isEmpty) return const <Post>[];

  return await ref.watch(feedForFriendsProvider(friendUids).future);
});
```

Three short-circuit conditions (each verified by a spec scenario):

| Condition | Result | Spec scenario |
|---|---|---|
| `auth == null` (unauthenticated) | `const <Post>[]` | SCENARIO-142 |
| `auth != null && friendUids.isEmpty` (no friends) | `const <Post>[]` | SCENARIO-141 |
| `auth != null && friendUids.isNotEmpty` (happy path) | result of `feedForFriendsProvider(friendUids)` | SCENARIO-140 |

**Why this composition avoids the `List<String>` equality trap** (REQ-FEED-PROVIDER-001 SCENARIO-143): the `friendUids` list is built **inside** the provider closure from an already-cached upstream (`acceptedFriendsProvider`). The same upstream emits the same `List<String>` instance until the underlying friendships change. Riverpod's family cache keys on identity-or-equality of the parameter — because we only call `feedForFriendsProvider(friendUids)` with the cached list (not a freshly constructed one per build), the family cache is hit, not invalidated.

The UI side never constructs a `List<String>` — it only calls `ref.watch(myFriendsFeedProvider)` which has no family parameter.

### 4.3 `feedSegmentProvider`

```dart
final feedSegmentProvider = StateProvider<FeedSegment>(
  (ref) => FeedSegment.amigos,
);
```

- Default is `FeedSegment.amigos` (SCENARIO-138).
- Only `FeedSegmentPills` writes (`ref.read(feedSegmentProvider.notifier).state = FeedSegment.amigos`). MI GYM and PÚBLICO never write (`onTap: null` for those pills in §3.3).
- `FeedScreen` reads via `ref.watch(feedSegmentProvider)` exactly once at the top of `build`.

### 4.4 `FeedScreen.build` — normative shape

```dart
@override
Widget build(BuildContext context, WidgetRef ref) {
  final segment = ref.watch(feedSegmentProvider);

  return Padding(
    padding: const EdgeInsets.symmetric(horizontal: 20),
    child: Column(
      crossAxisAlignment: CrossAxisAlignment.stretch,
      children: [
        const SizedBox(height: 20),
        const _FeedHeader(),
        const SizedBox(height: 14),
        const FeedSegmentPills(),
        const SizedBox(height: 14),
        Expanded(
          child: switch (segment) {
            FeedSegment.amigos => const _AmigosBody(),
            FeedSegment.gym || FeedSegment.public => const SizedBox.shrink(),
          },
        ),
      ],
    ),
  );
}
```

Note the **switch expression** with `||` pattern. Dart 3.0+ supports this. It compresses gym + public into a single `SizedBox.shrink()` branch, matching the spec scenarios (SCENARIO-148 + 149).

`_AmigosBody.build` reads `myFriendsFeedProvider` and uses `.when(...)` — see §3.4 for the tree.

### 7.1 New field declarations

`lib/features/feed/domain/post.dart` — final shape:

```dart

import 'package:freezed_annotation/freezed_annotation.dart';

import '../../profile/data/timestamp_converter.dart';
import 'post_privacy.dart';
import 'routine_tag.dart';

part 'post.freezed.dart';
part 'post.g.dart';

@freezed
class Post with _$Post {
  const factory Post({
    required String id,
    required String authorUid,
    // Author display fields denormalized at write time (same ADR as authorGymId).
    // Stale-on-update is accepted — standard social-media pattern.
    required String authorDisplayName,
    required String? authorAvatarUrl,
    required String? authorGymId,
    required String text,
    required RoutineTag? routineTag,
    required PostPrivacy privacy,
    @TimestampConverter() required DateTime createdAt,
  }) = _Post;

  factory Post.fromJson(Map<String, Object?> json) {
    final processed = Map<String, Object?>.from(json);
    if (processed['authorDisplayName'] == null) {
      processed['authorDisplayName'] = 'Anónimo';
    }
    // Note: authorAvatarUrl is nullable on the model. Missing key → fromJson assigns null
    // automatically because the generated factory handles nullable fields. Do not pre-set it.
    return _$PostFromJson(processed);
  }
}
```

Field order (for predictable freezed regen + readable JSON):
1. `id`
2. `authorUid`
3. `authorDisplayName` (new)
4. `authorAvatarUrl` (new)
5. `authorGymId`
6. `text`
7. `routineTag`
8. `privacy`
9. `createdAt`

The one-line comment block above the two new fields mirrors REQ-FEED-POST-001 requirement verbatim.

### 7.2 `fromJson` resilience — approach decision

**Picked approach (a)**: override the generated `fromJson` factory with a manual one that pre-processes the map (defaults missing `authorDisplayName` to `'Anónimo'`), then delegates to `_$PostFromJson`.

**Why not approach (b)** (`@Default('Anónimo')`): the freezed `@Default` annotation only fires when the **field** is not provided to the constructor — it does NOT cover the JSON-deserialization path when the field is absent. Worse, `@Default` requires the field to be non-`required`, which means making `authorDisplayName` an optional named parameter, which silently allows tests and the repo to construct `Post(...)` instances without specifying a display name. That is exactly the bug the resilience scenarios (134, 137) are trying to prevent at compile time.

Approach (a) keeps `authorDisplayName` as `required` (compile-time enforcement) AND provides runtime fallback for missing JSON keys (Firestore docs written before the amendment). Best of both.

The trade-off: when the generated `_$PostFromJson` is regenerated, the manual override stays untouched (it lives in the human-written part of `post.dart`, not `post.g.dart`). `build_runner` regeneration is safe.

### 8.5 Spec risk resolution — concrete picks

The spec called out 3 risks where the design must lock the choice:

#### Risk A — Relative timestamp format

**Pick**: simple Spanish relative time, lowercase "hace" prefix.

| Delta from now | Format | Example |
|---|---|---|
| < 1 minute | `'recién'` | recién |
| ≥ 1 min, < 60 min | `'hace ${m}m'` | hace 5m |
| ≥ 1 hour, < 24 hours | `'hace ${h}h'` | hace 2h |
| ≥ 1 day, < 7 days | `'hace ${d}d'` | hace 3d |
| ≥ 7 days | `'DD/MM'` (using `intl.DateFormat('dd/MM')`) | 08/05 |

The spec scenario SCENARIO-168 uses a case-insensitive regex `r'[Hh]ace\s+\d+\s*h'` — our format satisfies it.

Implementation: a private top-level function `_relativeTime(DateTime createdAt)` inside `post_card.dart` (no new package, no `timeago` dependency).

```dart
String _relativeTime(DateTime createdAt) {
  final delta = DateTime.now().difference(createdAt);
  if (delta.inMinutes < 1) return 'recién';
  if (delta.inHours < 1)   return 'hace ${delta.inMinutes}m';
  if (delta.inDays < 1)    return 'hace ${delta.inHours}h';
  if (delta.inDays < 7)    return 'hace ${delta.inDays}d';
  // ≥ 7 days: numeric short date
  return '${createdAt.day.toString().padLeft(2, '0')}/${createdAt.month.toString().padLeft(2, '0')}';
}
```

No `intl` dependency needed — manual `padLeft` for the date branch keeps `package:intl` out of the diff.

#### Risk B — `_RoutineTagChip` tap test harness

**Pick**: `GoRouter` mock (mirrors `routine_card_test.dart:94-129` exactly). NOT `MockNavigator`.

Why: `routine_card_test.dart` is the canonical precedent in this codebase. `context.push` works through the `GoRouter` provided via `MaterialApp.router(routerConfig: ...)`. Mocking the `GoRouter` itself is awkward (it has many internal classes); building a minimal real `GoRouter` with two routes (`/` and `/workout/routine/:id`) and asserting the body text changes is the idiomatic pattern.

Test outline for SCENARIO-171:

```dart
testWidgets('SCENARIO-171: tapping routine chip navigates to /workout/routine/:id', (tester) async {
  final post = makePost(routineTag: const RoutineTag(routineId: 'r1', routineName: 'Push · Día 4'));
  await tester.pumpWidget(_wrapRouter(PostCard(post: post)));
  await tester.pump();
  await tester.tap(find.text('Push · Día 4'));
  await tester.pumpAndSettle();
  expect(find.text('detail-r1'), findsOneWidget);
});
```

#### Risk C — Opacity assertion

**Pick**: plain `Opacity` widget (NOT `AnimatedOpacity`).

Why: no animation is needed — the pill state is static for Etapa 2 (MI GYM/PÚBLICO never become enabled). `AnimatedOpacity` would add 200ms transition flicker for no UX benefit. Using `Opacity` makes the spec scenarios (SCENARIO-161, 162) trivially assertable via:

```dart

  of: find.text('MI GYM'),
  matching: find.byType(Opacity),
);
final opacityWidget = tester.widget<Opacity>(opacityFinder);
expect(opacityWidget.opacity, equals(0.4));
```

If `AnimatedOpacity` were used, the assertion would need to settle the animation first and the test would be brittle to timing.

## 9. Edge case decisions

### Decision 1 — Loading skeleton vs spinner

**Choice**: spec mandates `CircularProgressIndicator(color: palette.accent)` centered (REQ-FEED-SCREEN-004). **Lock here**: no skeleton cards in Etapa 2.

**Why**: matches `PlantillasSection` pattern. Skeleton cards are polish, not scope. The header + pills are NOT covered by the loading indicator — they render regardless because they're hardcoded UI. Only the AMIGOS body slot shows the spinner.

**Implementation**: `Center(child: CircularProgressIndicator(color: palette.accent))` inside the `_AmigosBody` `.when(loading: ...)` branch.

### Decision 2 — Empty state copy

**Choice**: `'Aún no hay posts de tus amigos'` (exact string, no trailing period).

**Why**: matches REQ-FEED-EMPTY-001 SCENARIO-185 verbatim. Tone is warm/inviting (no "no friends found" cold feel). No CTA button below the text (Etapa 5 adds "Buscar amigos" wire when `/feed/search` exists). Layout per §3.7.

### Decision 3 — Initials algorithm

**Choice**: same as `HomeHeader._computeInitials` but with one extra branch for `'Anónimo'`.

```dart
String _computeInitial(String displayName) {
  if (displayName.isEmpty || displayName == 'Anónimo') return '?';
  return displayName.characters.first.toUpperCase();
}
```

- `'Tincho'` → `'T'`
- `'Sofía R.'` → `'S'` (capital S, dotted i not normalized)
- `'martín'` → `'M'` (capital M)
- `'Anónimo'` → `'?'` (the fromJson fallback marker — treated as unknown for avatar purposes)
- `''` → `'?'`

Single letter, not "MR" two-letter (consistent with home-shell ADR-HS-6). `.characters.first` (from `package:characters`, transitive) handles multi-code-point graphemes safely.

### Decision 4 — Gym ID display

**Choice**: render raw `post.authorGymId` uppercased.

**Why**: gym name resolution would require a new `gymsCollection` provider — out of scope per propose. The seed gym IDs (`seed_gym_001`, `seed_gym_002`, `seed_gym_003`) look ugly uppercased, but real production gym IDs are slugs like `la-fuerza-recoleta` which uppercase cleanly into `LA-FUERZA-RECOLETA`. Fase 4 (workout/gym integration) can wire a `gymNameProvider`.

Formatter: inside `_formatMeta` (§3.5):

```dart
final gym = (gymId == null || gymId.isEmpty) ? null : gymId.toUpperCase();
```

If `authorGymId` is null/empty, the meta line is just the timestamp (`'hace 2h'`). No leading separator.

### Decision 5 — Relative timestamp formatter

**Choice**: inline helper inside `post_card.dart`, no new package (§8.5 Risk A).

**Why**: `timeago` would add a dep for a 6-line function. `intl` is heavier; we only need short date format which is trivial with `padLeft`. The formatter is small enough to live inside the file that uses it — no need to extract.

### Decision 6 — `PostCard` verified badge

**Choice**: **do NOT render the verified badge in Etapa 2**.

**Why**: the `Post` model has no `authorVerified` field. The mockup shows the badge on the second card (Sofia R.) because that's the design intent for verified users, but there is no data source in this PR to drive it. Adding a `Post.authorVerified` field would be feature creep (the propose locked exactly 2 new fields: `authorDisplayName` + `authorAvatarUrl?`). The `TreinoIcon.verified` constant is still added (REQ-FEED-ICON-001) for future use — Etapa 4 or Fase 4 can wire it when verified status arrives.

Spec consequence: no scenario asserts the verified badge is rendered. SCENARIO-189 only checks `TreinoIcon.verified` is a valid `IconData`. No regression risk.

### Decision 7 — Stats stub: exact icons + values

**Choice**: NO icons, just three em-dash strings: `'— kg'`, `'— min'`, `'— ej.'`.

**Why**: REQ-FEED-POSTCARD-004 mandates em-dashes (not zeroes) and requires 3 stub values. Adding stat icons (e.g. `TreinoIcon.tabWorkout` for "ej.", `TreinoIcon.clock` for "min") would imply the data exists — em-dashes alone communicate "stat unknown" clearly. Fase 4 wires real numbers AND adds icons in a single change.

Inline comment in source per spec: `// Stub: real stats wired in Fase 4.`

### Decision 8 — `TreinoIcon.dotsThree` and `TreinoIcon.verified` Phosphor constants

**Choice**:
- `TreinoIcon.dotsThree` → `PhosphorIconsRegular.dotsThreeVertical`
- `TreinoIcon.verified` → `PhosphorIconsFill.sealCheck`

**Why**: `dotsThreeVertical` is the post overflow standard (vertical orientation matches the mockup). `sealCheck` (fill variant) is the conventional verified-account icon across Phosphor's iconset and matches the mockup's small filled checkmark. Both constants are namespaced under `TreinoIcon.X` so no `PhosphorIcons.*` direct usage appears in any new feed file (REQ-FEED-ICON-001 constraint).


### Decision 9 — Disabled pill behavior (MI GYM + PÚBLICO)

**Choice**: `Opacity(opacity: 0.4, child: _Pill(label: '...', isActive: false, onTap: null))`.

**Why**: simplest implementation that satisfies REQ-FEED-PILLS-003 (visual opacity 0.4) AND REQ-FEED-PILLS-004 (tap is no-op). `_Pill` accepts `VoidCallback? onTap`; when null, its internal `GestureDetector(onTap: null)` becomes a no-op. The outer `Opacity` dims the entire pill subtree (border, fill, text — exactly what the spec asserts).

**Rejected alternatives**:
- `IgnorePointer + Opacity`: redundant. `GestureDetector(onTap: null)` already ignores pointer events.
- `AnimatedOpacity`: brittle for tests; no animation needed (§8.5 Risk C).
- `Container.color.withValues(alpha: 0.4)`: would dim only the background, not the border/text. Fails the SCENARIO-161 finder.
